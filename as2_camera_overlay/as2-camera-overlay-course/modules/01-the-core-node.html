The Core Node

Meet OverlayNode, the coordinator of the entire system. It acts like a movie director, making sure the camera, the actors (3D markers), and the stage (the graphics engine) are all in sync.

💡
Key Insight

By inheriting from rclcpp::Node, this class becomes a citizen of the ROS 2 network, capable of listening to camera feeds and publishing the final AR video.

The Constructor & Lifecycle

When the node starts, it follows a strict sequence to set up the environment. It uses a wall timer to defer complex initialization until the object is fully ready.

CODE 

OverlayNode::OverlayNode(const rclcpp::NodeOptions &options) {
  declareParameters();
  tf_buffer_ = std::make_shared<tf2_ros::Buffer>(this->get_clock());
  initRenderer();
  init_timer_ = this->create_wall_timer(std::chrono::nanoseconds(0), [this]() {
    loadDisplays();
    startAttached();
  });
}
PLAIN ENGLISH

Start the node with user-provided options...

Read settings (topics, zoom) from the config file...

Prepare the "Spatial Memory" (TF) to track the drone's position...

Boot up the 3D graphics engine (Ogre)...

Wait for one microsecond before finishing setup...

Load the 3D markers (Grids, Paths) into the scene...

Finally, subscribe to the real camera feeds.

Finding the Camera

To draw markers correctly, the node needs to know exactly where the camera was in the world when the photo was taken. lookupCameraPose asks the spatial system for these coordinates.

The Main Loop

Every single camera frame goes through renderAndPublish. It handles rate limiting, image scaling for performance, and the actual rendering.

⏱️
Rate Limiting (max_render_fps)

Skips frames if the computer is too slow to keep up.

⚖️
Scaling (render_scale)

Shrinks the image temporarily to draw markers faster.

🚀
Publishing

Sends the final augmented image back to the drone network.
