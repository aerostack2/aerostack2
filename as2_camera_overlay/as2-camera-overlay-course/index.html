The Core Node

Meet OverlayNode, the coordinator of the entire system. It acts like a movie director, making sure the camera, the actors (3D markers), and the stage (the graphics engine) are all in sync.

💡
Key Insight

By inheriting from rclcpp::Node, this class becomes a citizen of the ROS 2 network, capable of listening to camera feeds and publishing the final AR video.

The Constructor & Lifecycle

When the node starts, it follows a strict sequence to set up the environment. It uses a wall timer to defer complex initialization until the object is fully ready.

CODE 

OverlayNode::OverlayNode(const rclcpp::NodeOptions &options) {
  declareParameters();
  tf_buffer_ = std::make_shared<tf2_ros::Buffer>(this->get_clock());
  initRenderer();
  init_timer_ = this->create_wall_timer(std::chrono::nanoseconds(0), [this]() {
    loadDisplays();
    startAttached();
  });
}

Projecting the Video

The drone's video isn't just "under" the 3D markers—it's actually part of the 3D scene. We draw it on a flat panel (a Quad) that fills the camera's view perfectly.

CODE 

void OverlayRenderer::createBackgroundQuad() {
  background_rect_ = new Ogre::Rectangle2D(true);
  background_rect_->setCorners(-1.0, 1.0, 1.0, -1.0);
  background_rect_->setRenderQueueGroup(Ogre::RENDER_QUEUE_BACKGROUND);
  Ogre::AxisAlignedBox aab_inf;
  aab_inf.setInfinite();
  background_rect_->setBoundingBox(aab_inf);
}
PLAIN ENGLISH

Create a 2D rectangle that will hold our video pixels...

Make it fill the entire screen (from top-left to bottom-right)...

Tell the engine to draw this rectangle FIRST, behind everything else...

Define a bounding box for the rectangle...

Make the box "Infinite" so the engine never hides the background, no matter where the camera looks.

Fake Environment

Standard RViz plugins are "spoiled"—they expect to live in a fancy windowed application with a mouse and a screen. Since we are running headlessly, we have to provide "Fake" versions of the environment they expect.

The Headless Cast Classes that trick RViz plugins
HeadlessRosNode Wraps the real ROS node to hide the headless nature.
HeadlessFrameManager Handles coordinate requests (TF) headlessly.
NoOpHandlerManager Safely ignores mouse clicks and keyboard events.
HeadlessDisplayContext The master "Fake RViz" shell.

The Master Illusionist

HeadlessDisplayContext is the most important class in this module. It implements the standard RViz DisplayContext interface but returns nullptr (Nothing) for any window-related request.

CODE 

rviz_common::WindowManagerInterface *
HeadlessDisplayContext::getWindowManager() const {
  return nullptr; 
}

void HeadlessDisplayContext::queueRender() {
  render_dirty_.store(true);
  frame_count_.fetch_add(1);
}

End of the window manager function.

When a plugin changes a setting and wants the screen to refresh...

We flip a "Dirty" switch to true, signaling that the next frame needs a redraw...

We increment the frame counter so the plugin knows time is moving.

Ignoring the Mouse

RViz plugins allow users to click 3D objects to see data. This uses a SelectionManager. Since we have no mouse, NoOpHandlerManager provides empty slots where plugins can "register" their click-handlers without crashing the program.

ℹ️
Architecture Detail

This "No-Op" pattern is a common trick in software engineering. By providing an object that does nothing instead of "No Object" (null), you prevent "Null Pointer Exceptions" and keep the code running smoothly.

Plugin Loader

The "Secret Sauce" of as2_camera_overlay is that it doesn't try to reinvent the wheel. It uses a plugin loader to use the exact same code that RViz uses to draw grids and markers.

💡
Key Insight

By using pluginlib, we can load any RViz plugin dynamically. If a new type of marker is created for RViz, our node can use it instantly without any code changes!

The Plugin Manager

DisplayLoader is our dynamic manager. It takes a "Class ID" (like rviz_default_plugins/Grid) and instantiates the plugin code on the fly.

CODE 

bool DisplayLoader::loadDisplay(const DisplayConfig &cfg, 
                                const YAML::Node &properties) {
  auto display = loader_->createSharedInstance(cfg.class_id);
  display->initialize(context_);
  applyProperties(display->getRootProperty(), properties, logger_);
  display->setEnabled(true);
  displays_.push_back(display);
}

Ask the plugin engine (loader) to create a copy of the plugin code...

Give the plugin our "Fake RViz" environment to live in...

Translate the user's YAML settings into the plugin's internal properties...

Turn the plugin ON so it starts listening to data...

Save the plugin in our list of active actors.

Math Utilities

Drawing 3D markers on a 2D image is all about projection math. If the math is wrong by even 1%, the markers will appear to "drift" or float away from the drone's actual position.

💡
The "Secret Hack"

The 4x4 projection matrix is the most critical piece of math in the project. It warps the virtual 3D world to exactly match the distortions of the real physical camera lens.

The Lens Blueprint (Intrinsics)

Every camera has a "personality"—its focal length, its center point, and its sensor size. We capture this in the Intrinsics struct.

Correcting Coordinate Chaos

Computer Vision and 3D Graphics don't speak the same language. visionToOgreRotation acts like a translator, swapping axes so that "Forward" in one system is "Forward" in the other.

1
CV Frame

Z is forward, X is right, Y is down.

2
Ogre Frame

Z is back, X is right, Y is up.

3
The Fix

Rotate the coordinates by 90/180 degrees to align them perfectly.

Stereo Offset (The Eyes of the Drone)

If the drone has two cameras (Stereo), applyStereoBaseline shifts the 3D world slightly left or right to match which "eye" is currently being used. Without this, the AR overlay would look "cross-eyed"!
