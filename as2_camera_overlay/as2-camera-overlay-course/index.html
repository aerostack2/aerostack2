01

Headless AR

Augmented Reality for drones — even when nobody is watching

The Problem: "Eyes" without a Brain

Imagine you're flying a drone autonomously. The drone sees the world through a video feed. But the drone's brain is also calculating things you *can't* see—like where it thinks the landing pad is, or the path it plans to take.

Usually, engineers use a GUI tool called RViz to see these hidden details. But drones have tiny computers with no screens attached.

💡
The "Invisible Ink" Solution

as2_camera_overlay is like a special highlighter. It takes those invisible 3D thoughts from the drone's brain and "paints" them directly onto the camera's pixels before the video is even sent.

A Conversation in Code

Inside the drone, different parts of the software are constantly talking to each other using messages.

The Three Roles

In the codebase, work is split between three main roles. Each one has a specific "superpower."

🤖

The Coordinator

OverlayNode acts like a project manager. It listens for camera images and hands them off to the artist.

🎨

The Artist

OverlayRenderer uses a 3D engine to paint markers and grids onto the video pixels.

🔌

The Plugin Manager

DisplayLoader lets us add new types of visuals without changing the main core code.

Step 3: Finding the Drone

This is the most critical step. We need to know exactly where the camera is in the world to draw the markers in the right place.

CODE 

if (!lookupCameraPose(camera_frame_id, header.stamp, cam_pos, cam_rot)) {
  return;
}

renderer_->setIntrinsics(scaled_k, near_plane_, far_plane_, zoom_factor_);
renderer_->setCameraPose(cam_pos, cam_rot);
renderer_->updateBackgroundImage(*background_image);

rendered = renderer_->renderAndRead();
PLAIN ENGLISH

Ask the TF library exactly where the camera was when the photo was taken.

If we don't have a pose for the camera, we can't draw anything accurately, so we skip this frame.

Tell the virtual camera about its lens properties (called intrinsics).

Move the virtual camera to match the real camera's position.

Paste the real camera image behind our 3D scene.

Take a 'snapshot' of the finished work and prepare to share it.

The "Aha!" Moment

💡
Key Insight: The Virtual Twin

This code works by creating a "digital twin" of the drone's camera. By moving a virtual camera in the same way the real camera moves, the 3D markers line up perfectly with the real-world objects in the video.

04

Clever Tricks

How to make 3D rendering work on a flying robot

The RViz Cheat Code

Drawing 3D markers like grids and paths is hard. Instead of writing that code from scratch, the developers used a clever trick: they reused the exact same plugins that RViz uses.

🔌

Reuse, Don't Rewrite

Because it uses standard plugins, any 3D marker that works in RViz will work here too. It's like being able to use any brand of headphones with a phone because they all use the same plug.

🖼️

Virtual Buffer

The code tricks the graphics engine into rendering into a buffer instead of a window on a screen.

The Performance Hack: Render Scale

Drawing high-definition 3D graphics on a drone takes a lot of GPU power. To save energy and keep things fast, the developers added a "Scale" trick.

CODE 

scaled_k.width = std::max(1u, static_cast<unsigned int>(orig_w * render_scale_));
scaled_k.height = std::max(1u, static_cast<unsigned int>(orig_h * render_scale_));

cv::resize(cv_ptr->image, scaled_bg, cv::Size(scaled_k.width, scaled_k.height), 0, 0, cv::INTER_AREA);

// ... (after rendering) ...

cv::resize(rendered, rendered, cv::Size(orig_w, orig_h), 0, 0, cv::INTER_LINEAR);
PLAIN ENGLISH

Shrink the image size (e.g., to 50%) before we start drawing. This makes the math much easier for the drone.

Shrink the background photo to match our smaller drawing space.

The GPU now has 4x fewer pixels to worry about, making it much faster.

After we're done drawing, stretch the image back up to its original size so the user doesn't notice.

This trick is essential for maintaining a high frame rate while the drone is flying fast.
